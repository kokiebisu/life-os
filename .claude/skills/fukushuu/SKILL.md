# 復習 - 忘却曲線ベーススペーシドリピティション

`aspects/study/` の学習ノートを忘却曲線に基づいて復習する。

**対象ディレクトリ:** `aspects/study/`（software-engineering、interview-prep など全サブディレクトリ）
**対象外:** `aspects/reading/`（読書記録は含めない）

## 復習間隔

| 累計復習回数 | 次回まで |
|---|---|
| 未復習 | 即座に対象 |
| 1回目後 | 1日 |
| 2回目後 | 3日 |
| 3回目後 | 7日 |
| 4回目後 | 14日 |
| 5回目後 | 30日 |
| 6回目以降 | 60日 |

判定は3段階: ⭕ 完璧（+1）、🔺 あいまい（据え置き）、❌ 忘れた（リセット・1日に戻す）（Claude が面接官として判定）。詳細は Step 4 参照。

## Step 1: 今日の日付を確認

```bash
TZ=Asia/Tokyo date +%Y-%m-%d
```

## Step 2: 復習ログを読む + マイグレーション

`aspects/study/review-log.json` を Read する。ファイルがなければ空の `{}` として扱う。

### マイグレーション（厳守・冪等）

各エントリに `pending_questions` フィールドが無ければ `[]` を追加して書き戻す。これは冪等処理なので毎回実行してよい。

```typescript
for (const [path, entry] of Object.entries(log)) {
  if (!('pending_questions' in entry)) {
    entry.pending_questions = [];
  }
}
// 変更があった場合のみ Write で書き戻す
```

書き戻した後、後続の Step に進む。

## Step 3: 対象ノートを特定

### Step 3-1: 候補プールを構築

Glob で `aspects/study/**/*.md` を取得し、以下を**除外**する:

- `**/CLAUDE.md`
- `**/README.md`
- `**/roadmap.md`
- `**/tracker.md`（進捗管理・10日スケジュール等のメタ文書）
- `**/qa-bank.md`（独自Q&Aバンクは別機構で運用）
- `**/system-design-chapters.md`（Chapter優先度マップ・メタ参照）

残ったノートを以下の2カテゴリに分類する:

- **面接対策:** `aspects/study/interview-prep/**`
- **読書（本のノート）:** それ以外（`aspects/study/software-engineering/**` など）

### Step 3-2: カテゴリ選択（ユーザーに尋ねる）

**動的検出（実行時に毎回評価する）:**

- **書籍**: `aspects/study/*/*/` のうち `ch*.md` を含むディレクトリ（例: `software-engineering/システム設計の面接試験/`）
- **トップレベルトピック**: `aspects/study/<dir>/` のうち、`ch*.md` を含まず、`interview-prep/` `team/` 以外のディレクトリ（例: `algorithm/`, `sql/`）
- **面接対策トピック**: `aspects/study/interview-prep/<company>/` 直下の `.md` ファイル（`prep.md` `qa-bank.md` `journey.md` を除く）。例: `resilire/db-study.md`, `resilire/go-study.md`, `resilire/system-design.md`

#### 第1階層メニュー

AskUserQuestion で以下の選択肢を提示する（現時点で期日が来ているノートがある項目のみ表示）:

- **滞留集中ドリル（forgot N / fuzzy M）** — `confidence ∈ {forgot, fuzzy}` かつ `review_count == 0` で停滞しているノート。**N + M ≥ 3 のときだけメニューに出し、(Recommended) を付ける**。閾値未満なら表示しない
- **バランス（おまかせ）** — 面接対策 と 読書 を均等に抜粋（滞留ドリルが非表示のときの Recommended）
- **面接対策のみ** — `aspects/study/interview-prep/**` のノートだけ
- **書籍別** — 章ノート（書籍）から選ぶ。選択時に第2階層メニューを開く
- **トピック別** — トピックノート（algorithm / sql / 面接対策技術ガイド等）から選ぶ。選択時に第2階層メニューを開く

各項目の末尾に**期日が来ているノート件数**を `（全N件中M件）` で表示する（例: `バランス（おまかせ）（全34件中5件）`）。書籍別 / トピック別の件数は配下の合算。滞留ドリルは `（forgot N / fuzzy M）` の内訳のみ。

#### 第2階層メニュー（書籍別 / トピック別を選んだとき）

選択肢は最大4件。**先頭に必ず「おまかせ」を Recommended として置く**。残りは期日が来ているノートを多く持つ順に並べ、4枠を超える場合は「その他（手動指定）」を最後に置く。

- **書籍別を選んだ場合**:
  - 1枠目: **おまかせ（全書籍）** — 全書籍の章ノートから期日が古い順に最大5件 (Recommended)
  - 残り: 各書籍を `[書籍名]（全N件中M件）` 形式で表示
- **トピック別を選んだ場合**:
  - 1枠目: **おまかせ（全トピック）** — 全トピックノートから期日が古い順に最大5件 (Recommended)
  - 残り:
    - **トップレベルトピック** は `[ディレクトリ名]（全N件中M件）` 形式（例: `algorithm（全4件中4件）`）
    - **面接対策トピック** は `[company]/[topic]（全N件中M件）` 形式（例: `resilire/db-study（全1件中1件）`）

「その他」が選ばれたら、ユーザーに対象トピック名/書籍名を直接入力させる。

### Step 3-3: 期日判定とフィルタリング

各カテゴリ内で期日判定（review-log.json と照合）を行い:

```
interval_map = { 0: 0, 1: 1, 2: 3, 3: 7, 4: 14, 5: 30, それ以降: 60 }
due = (last_reviewed ≠ today) かつ (last_reviewed + interval_days ≤ today)
      または (ログになければ即対象)
```

**同日二重復習の禁止（厳守）:** `last_reviewed == today` のエントリは対象外。fuzzy で据え置きになっても、同じ日に再出題してはいけない。

ユーザーの選択に応じて候補プールを絞る:

- **滞留集中ドリル** → `confidence ∈ {forgot, fuzzy}` かつ `review_count == 0` のノートのみ。`last_reviewed` が古い順に**最大5件**。期日判定（due）はスキップ（停滞ノートは定義上「ずっと due」だが、判定式を介さず直接抽出する）。判定・更新フロー（Step 4 / Step 5）は他モードと同じ — ⭕ なら `review_count: 1, interval_days: 1` に進めて停滞から脱出させる
- **バランス** → 面接対策と読書を均等に抜粋、**合計最大5件**（例: 面接対策3件 + 読書2件）。どちらかに対象がなければ、もう一方から補う
- **面接対策のみ** → interview-prep ノートから期日が古い順に**最大5件**
- **特定書籍** → その書籍の章ノートから期日が古い順に**最大5件**
- **おまかせ（全書籍）** → 全書籍の章ノートから期日が古い順に**最大5件**
- **おまかせ（全トピック）** → 全トピックノート（トップレベルトピック + 面接対策トピック）から期日が古い順に**最大5件**
- **トップレベルトピック**（algorithm / sql 等）→ そのディレクトリ配下の全 `.md` から期日が古い順に**最大5件**
- **面接対策トピック**（resilire/db-study 等）→ そのファイル単体（1件）。今後同名ディレクトリ配下に派生ノートが増えたらそれも対象に含める

**優先順位（厳守）:** 未復習ノート（review-log.json にエントリがないもの）を最優先する。既存ノートの期日到来より未復習ノートが常に優先される。未復習を全て処理してから、再復習ノートを期日が古い順に追加する。

対象ノートが0件の場合は「今日復習すべきノートはありません ✅ 次の復習は〇〇日（最も近い期日）です」と伝えて終了。

## Step 4: 対話的クイズ

対象ノートを1件ずつ処理する。

### 出題形式の切り替え（厳守）

技術面接の本体は実装・コードレビュー・クエリ作成。**説明だけ問う出題に偏らないこと。** ノートのカテゴリ（`category` frontmatter or ディレクトリ）で出題形式を切り替える:

| カテゴリ | 出題形式 |
|---|---|
| `algorithm` | **実装系**を優先: 「ノート見ずに `〇〇関数` を Go で書いて」「LC の似た問題を解いて」「このコードのバグを直して」 |
| `sql` | **クエリ系**: 「このスキーマで〇〇を取得するクエリを書いて」「このクエリの結果を予測して」「LEFT JOIN なのに INNER 挙動になる原因を直して」 |
| `interview-prep`（コーディング/コードレビュー系） | **実装/レビュー系**: コーディング題材なら再実装、コードレビュー題材なら指摘の再現 |
| `system-design` / `database-design` 等の概念系 | **概念確認 + 応用ミックス**（説明系もOK、ただし「面接で口頭で2分話せるか」の粒度） |

既存の `❓ 自分への質問` が説明型ばかりだった場合でも、**Claude が判断して実装系に切り替えてよい**（既存キューに縛られない）。ユーザーが「説明モードで」と明示した場合のみ、既存キューをそのまま使う。

### 出題プロンプトの禁止事項（厳守）

質問文だけを出す。**前回の詰まり要約・正解ポイント・ヒント類はプロンプトに含めない。** 具体的には:

- ❌「前回〇〇で詰まったので、今回はそれも含めて答えて」
- ❌「前回 X と回答（正: Y）の点を踏まえて」
- ❌ ノートの `🔁 復習で詰まったところ` セクションから要約を生成してプロンプトに付け足す
- ❌ review-log.json の `pending_questions` 履歴から「なぜ前回詰まったか」を要約して添える

詰まり履歴は**判定後の解説**でだけ使う（ユーザー回答後に「前回もここで詰まった」と振り返るのはOK）。

**Why:** 前回の詰まりポイントを出題時に開示すると、ほぼ答えそのものを伝えてしまい、自力で思い出すテストが成立しない。

### 各ノートの処理

1. ノートを Read する
2. タイトルのみを表示する。**`🧒 一言で言うと` は表示しない**（後続の質問の答えのキーワードを漏洩する可能性が高いため）
3. **pending_questions の確認（厳守）**: review-log.json の該当ノートエントリから `pending_questions` を取得する
   - **`pending_questions` が空でない場合**: そのリストの質問だけを 1 問ずつ出題する。ノートの `❓ 自分への質問` セクションは**参照しない**（前回詰まった部分だけに集中再出題する仕組み）
   - **`pending_questions` が空の場合**: 通常通り `❓ 自分への質問` を出題する（カテゴリに応じた角度切り替えも従来通り適用）
4. 上記の出題形式ルールに従い、**1問ずつ** 出題する。既存キューがある場合はベースにしつつ、カテゴリに合った実装/クエリ系に翻訳して出題してよい
   - ユーザーが答えるのを待つ
   - 答えが返ってきたら、ノートの該当箇所を引用して解説・補足する
   - 次の質問がある場合は続ける
5. 全質問の回答中、Claude が**面接官として**各質問を ✅ / ⚠ / ❌ の3段階で内部判定する（Step 4.5 で使用）:
   - ✅ 正解 — 主要概念を正確に答えた
   - ⚠ 部分正解 — 主要概念は合っているが細部の値・定義・基準を欠落
   - ❌ 詰まり — 主要概念を取り違えた / 「わからない」「忘れた」と即答

   全問終了後、Claude が判定結果を集約して全体評価を提示する（AskUserQuestion は使わない）:
   - 全質問が ✅ → ⭕ 完璧
   - ❌ なし、⚠ が1問以上 → 🔺 あいまい
   - ❌ が1問以上 → ❌ 忘れた

   提示フォーマット例:
   > 面接官として判定: **❌ 忘れた**
   > 理由: POA/DOA を逆に説明したため

   ユーザーが異議（「いや今のは正解」等）を出したら素直に再判定する。
6. 回答に応じて次回スケジュールを伝える + pending_questions を更新する:
   - ⭕ 完璧（全問 ✅） → review_count +1、通常間隔テーブル通り。**`pending_questions = []` にクリア**。「次回は〇〇日です」
   - 🔺 あいまい（⚠ あり、❌ なし） → review_count 変更なし、interval_days は `max(interval_days, 1)`（未復習状態の 0 は 1 に底上げ、それ以外は据え置き）、last_reviewed を今日に更新。**`pending_questions = ⚠ と判定された質問のテキストだけ`を保存**。「詰まった N 問を次回もう一度確認します。次回は〇〇日です」
   - ❌ 忘れた（❌ が1問以上） → review_count を 0 にリセット、interval_days を 1 に。**`pending_questions = ⚠ + ❌ と判定された質問のテキスト`を保存**。「もう一度、明日復習しましょう」

   **pending_questions 保存時の注意:**
   - 質問テキストは出題したそのままの文字列を保存する（要約・短縮しない）
   - 動的生成キュー（`❓ 自分への質問` セクションが無くて生成した質問）も同じく保存対象
   - 保存後、次回そのノートを復習するときは Step 4 の手順 3 の分岐で pending_questions が優先される

### コーネル式キューがないノートの場合

`❓ 自分への質問` セクションが存在しない、または空のとき:

1. **事前にキューを自動生成する**（ユーザーに確認しない）
   - ノート本文のセクション構造・箇条書きから、具体的な 3〜5問のキューを Claude が内部で生成する
   - 「何を学んだか説明して」のような抽象的な問いは禁止。必ず**具体的な値・定義・使い分け基準**を問う形にする
   - **カテゴリに応じた出題形式**を Step 4 の表に従って優先する（algorithm → 実装、sql → クエリ、概念系 → 応用ミックス）
   - 例（algorithm）: 「ノートの `lengthOfLongestSubstring` を見ずに Go で書いて」
   - 例（sql）: 「departments と employees から「部署ごとの社員数（0人含む）」を取得するクエリを書いて」
   - 例（概念系）: 「99.9% と 99.99% のダウンタイムを年間/月で言える？」「QPS 計算のピーク係数は何倍？」
2. 生成したキューを使い、**既存キューと同じフロー**で 1問ずつ出題する（タイトル表示 → 1問ずつ → 判定）
3. 判定の後、**ユーザーに確認なくノートに書き戻す**
   - 生成したキューを MD の末尾（または `💡 まとめ` の後）に `## ❓ 自分への質問（コーネル式キュー）` セクションとして追記する
   - Notion にも同期する（該当ノートが Notion にある場合）

## Step 4.5: 復習結果をノートに追記

ノートごとに、その回の判定結果を全体評価とともに必ず追記する（⭕ 完璧の場合も履歴として残す）。

### MD ノートへの追記

セクション位置: **ファイル末尾**（既存セクションの後）。

セクションが既に存在する場合:
- `## 🔁 復習で詰まったところ` の**直下**に新しい日付ブロックを挿入する（最新が上）

セクションが無い場合:
- ファイル末尾に `## 🔁 復習で詰まったところ` を新規作成し、最初の日付ブロックを追加

フォーマット:

`### YYYY-MM-DD - <全体評価>` を見出しに、評価ごとに本文を変える:

- **⭕ 完璧:** 単一行 `- 全問正解、詰まりなし` のみ
- **🔺 あいまい / ❌ 忘れた:** ⚠ または ❌ と判定された質問を Q ごとに `**Q: ...**` / `詰まった内容: ...` / `正解ポイント: ...` の構造で列挙

````markdown
## 🔁 復習で詰まったところ

### YYYY-MM-DD - ⭕ 完璧
- 全問正解、詰まりなし

### YYYY-MM-DD - 🔺 あいまい
- **Q: コーネル式キューの原文をそのまま記載**
  - 詰まった内容: ユーザー回答の要約（要点のみ。直接引用しすぎない）
  - 正解ポイント: ノート本文から該当箇所を1〜2行で抽出

### YYYY-MM-DD - ❌ 忘れた
- **Q: ...**
  - 詰まった内容: ...
  - 正解ポイント: ...
````

同一日付内の Q の順は出題順（古い → 新しい）。⭕ の場合は Q 列挙なし、単一行のみ。

### Notion ページへの追記

ノートのフロントマター `notion_id` を確認:

1. **`notion_id` あり:**
   - `notion-fetch` で既存本文を取得
   - 末尾に同じ Markdown ブロック（`## 🔁 復習で詰まったところ` の最新日付ブロック）を追記
   - `notion-update-page` の `replace_content` で全文書き戻す

2. **`notion_id` なし:**
   - `notion-add.ts --db study` 等で新規ページ作成
   - 取得した notion_id を md のフロントマターに書き戻す
   - 上記 1 と同じ手順で本文末尾に追記

   **例外:** interview-prep ノート（`aspects/study/interview-prep/**/note.md`）で notion_id がない場合は、対応するページが「勉強（面接対策）」DB（`33fce17f-7b98-8083-b569-d1e4bc70f867`）にあるか `notion-search` で確認してから判断する。見つかればその ID を frontmatter に書き戻して使う。見つからなければ新規作成。

### 失敗時

Notion 側の更新に失敗した場合、md は更新済みのまま **「Notion 同期失敗、再実行してください」** とユーザーに通知する。

## Step 5: review-log.json を更新

全ノートの処理が終わったら `aspects/study/review-log.json` を Write で更新する。

```json
{
  "aspects/study/<note-path>.md": {
    "last_reviewed": "YYYY-MM-DD",
    "interval_days": 3,
    "review_count": 2,
    "confidence": "perfect",
    "pending_questions": []
  }
}
```

- `last_reviewed`: 今日の日付（JST）。3段階すべてで更新する
- `review_count`: ⭕ 完璧 → +1、🔺 あいまい → 変更なし、❌ 忘れた → 0 にリセット
- `interval_days`: ⭕ 完璧 → interval_map から review_count に対応する値、🔺 あいまい → `max(interval_days, 1)`（0 は 1 に底上げ、それ以外は据え置き）、❌ 忘れた → 1
- `confidence`: Claude が面接官として下した最新の判定結果。`"perfect"` / `"fuzzy"` / `"forgot"` のいずれか。既存エントリに `confidence` がなくても正常動作する（後方互換）
- `pending_questions`: 前回詰まった質問テキストの配列。⭕ 完璧 → `[]` にクリア、🔺 あいまい → ⚠ 質問のみ、❌ 忘れた → ⚠ + ❌ 質問。Step 2 マイグレーションで全エントリに最低限 `[]` が入る

## Step 6: セッションサマリーを Notion「勉強（復習）」DB に書き出し（厳守）

review-log.json 更新後、**今日の復習セッション全体のサマリーを「勉強（復習）」DB に新規ページとして作成する。** 各ノート個別の追記（Step 4.5）とは別に、**セッション横断で見返せる1ページ**を必ず作る。

### Step 6-1: 「勉強（復習）」DB に新規ページ作成

- **DB**: `勉強（復習）` (database_id: `34fce17f-7b98-8012-905a-c024293a56f6`)
- **重複チェック**: 同日に既存エントリがある場合のみ、新規作成ではなく既存ページを `replace_content` で更新する。`notion-search` で `日付` が今日のエントリを探す
- **同日エントリがなければ新規作成**:

`notion-create-pages` を以下のプロパティで呼び出す:

| プロパティ | 値 |
|---|---|
| `名前` (title) | `復習セッション YYYY-MM-DD` |
| `日付` (date, datetime) | セッション開始 → 終了時刻（JST `+09:00` 必須） |
| `カテゴリ` (select) | ユーザーが Step 3-2 で選んだ値（`バランス` / `面接対策` / 書籍名 / トピック名のいずれか）。書籍名・トピック名は DB の select オプション一覧（`システム設計の面接試験` / `達人に学ぶDB設計徹底指南書` / その他）に存在しない場合は `その他` を使う |
| `件数` (number) | 復習対象ノートの総件数 |
| `結果` (rich_text) | 例: `⭕ 0 / 🔺 2 / ❌ 3`（完璧/あいまい/忘れた の件数集計） |
| `icon` | `✏️` |

### Step 6-2: ページ本文を `replace_content` で書き出し

以下のフォーマットを**必ず使う**。テンプレートを丸ごと埋め込む（プレースホルダ `{...}` を実値に置換）:

````markdown

- 「{選択カテゴリ}」選択で {内訳: 例「面接対策3件 + 書籍2件、すべて未復習ノート」}

|---|---|---|---|
| 1 | {ノート短縮タイトル} | {⭕ 完璧 / 🔺 あいまい / ❌ 忘れた} | {0 / N 問} |
| 2 | ... | ... | ... |

---

### {n}件目 {ノート短縮名} — {判定}

**詰まり{番号もしくはなし}: {テーマ要約}**
- {ユーザーの回答の誤り or 不足を要約}
- 正解: {ノート本文から該当箇所を引用、表・箇条書きを使って構造化}

**正解した質問:**
- {Q要約} ✅
- ...

（⭕ 完璧の場合は「全問正解、詰まりなし」のみ）

（全ノート分繰り返し）

- ...

（派生質問がなければこのセクションは省略）

## 🔑 キーワード

- {セッション全体で扱ったキーワードを箇条書き。各ノートのキーワード + 派生概念を統合}

## 💡 まとめ

1. **{N}件中 ❌ {a}件 + 🔺 {b}件 + ⭕ {c}件 = {次回スケジュール総括}**

3. **{特定の詰まり}** は次回必ず ... （次回フォーカスポイント）

---
````

### Step 6-3: ルール

- **テンプレ厳守**: 上記の見出し（`## 🎯` `## 📝` `## 🔑` `## 💡` `## ❓`）と順序は変えない。`派生質問` セクションは派生があるときのみ
- **判定アイコン**: ⭕ / 🔺 / ❌ を統一して使う
- **詰まりの構造**: 「**詰まり: テーマ**」→「ユーザー回答の誤り要約」→「正解: 構造化された解説（表・箇条書き）」
- **時間幅**: callout の時刻はページの `日付` プロパティ（start/end）から取る
- **新規作成は `notion-create-pages`**、同日既存エントリの更新は `notion-update-page` の `replace_content` を使う
- **JST タイムゾーン**: `日付` プロパティに時刻を入れる場合は必ず `+09:00` を付ける（`date:日付:start` / `date:日付:end` / `date:日付:is_datetime: 1`）
- **個別ノートの追記とは独立**: Step 4.5 で各ノートの md と Notion ページに追記した詰まり履歴は**そのまま残す**。Step 6 はそれとは別に、横断的に見返せる集約ページを作る役割

### Step 6-4: 終了報告

セッション終了時に、ユーザーへ以下のリンクと総括を提示:

> 📄 [今日の復習セッションサマリー](Notion URL)
>
> {N}件中: ⭕ {c} / 🔺 {b} / ❌ {a}
> 次回: {最も近い再復習日}

### Step 6-5: 失敗時

Notion 更新に失敗したら md は更新済みのまま **「セッションサマリー Notion 同期失敗、再実行してください」** とユーザーに通知する。

## 注意

- 1問ずつ出す。複数の質問を一度に表示しない
- ユーザーの答えを待つ前にノートの内容を開示しない
- セッション中に新しいノートが追加されていても対象に含める
- **面接官モード:** 判定は Claude が面接官として下す。基準は「面接で口頭説明した時に通用するか」。用語の取り違えは ❌、概念は合っているが細部精度が落ちるなら ⚠、スラスラ正確なら ✅
- ユーザーが判定に異議を出したら素直に再判定する。言い訳ではなく事実確認として扱う
