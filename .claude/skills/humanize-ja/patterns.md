# NG パターン詳細

[gonta223/humanizer-ja](https://github.com/gonta223/humanizer-ja) の 20 パターンをベースに、就活ドメイン特化のパターンを追加。原典は MIT License。

## 凡例

- **NG**: AI が好む典型句
- **置き換え方針**: 削るか、何で埋めるか
- **就活向け補足**: 履歴書・職務経歴書・面接文脈での注意

---

## カテゴリ A: humanizer-ja 流用

### A1. 意義の過剰強調

「重要な」「極めて」「計り知れない」などで物事を盛る。

**NG 例:** この取り組みは、業界全体の DX 推進において極めて重要な役割を果たしており、その意義は計り知れない。
**置き換え:** 具体的な数字・事実に置き換える。
**就活向け補足:** 「〜することで〇〇に大きく貢献」のような形が頻出。「〇〇を 70% 短縮」のような数値で代替する。

### A2. 定型評価語

| NG | 置き換え方 |
|---|---|
| 浮き彫りにしており | 具体的に何が見えたか書く |
| 今後の展開が注目されます | 削除するか、自分の予測を書く |
| 多面的な / 包括的な / 画期的な | 何がどう多面的なのか具体化 |
| 注目に値する | なぜ自分が注目したか 1 文で |
| 〜と言えるでしょう | 断定するか、削除する |
| 〜ではないでしょうか | 言い切る |
| 重要な示唆を与えている | 何の示唆か具体的に書く |

### A3. カタカナ語の過剰使用

「ソリューションをレバレッジして、イノベーティブなアプローチで」のような形。**置き換え:** 和語で言えるなら和語。カタカナが 3 語以上連続したら疑う。

**就活向け補足:** 技術用語（`Kubernetes`, `Terraform`, `FastAPI` 等）は固有名詞として使ってよい。一般概念のカタカナ化（「ソリューション」「アプローチ」）は和語化する。

### A4. 曖昧な出典

「業界の専門家によると」「調査結果が示すように」など。**置き換え:** 出典を出すなら具体的に（「Ahrefs が 90 万ページを分析したデータでは」）。出せないなら自分の意見として書く。

### A5. 太字 + コロンの箇条書き

```
- **速度:** 処理速度が 3 倍に向上
- **安全性:** セキュリティ基準を満たす
```

**置き換え:** ラベルを消して内容だけにする。

```
- 処理速度が 3 倍になった
- セキュリティ基準もクリア
```

**就活向け補足:** 職務経歴書ではこの形式が頻出するため最重要警戒。「【見出し】」形式や「〇〇について」前置きでも代替可。

### A6. 三点セットの強制

メリット 3 つ・ステップ 3 つに何でもまとめる。**置き換え:** 本当に 3 つなら 3 つでいい。4 つあるなら 4 つ書く。1 つだけでもいい。

### A7. 全角ダッシュ（——）の乱用

**NG:** Claude Code——Anthropic が開発した CLI ツール——を使えば
**OK:** Claude Code（Anthropic の CLI ツール）を使えば

**就活向け補足:** カッコ or 読点に置き換える。voice 参照（`voice.md`）の `**bold** — 補足` 構造での `—`（em dash）は OK。`——`（ダブル em dash）が NG。

### A8. 見出しの過剰構造化

300 字以下のセクションに小見出しは不要。

### A9. 説教くさい前置き

「ここで重要なのは〜という点です」「〜について理解しておく必要があります」「注意すべき点として」**置き換え:** 削除して内容をそのまま書く。

### A10. 接続詞の多用

「一方で」「しかしながら」「加えて」「このように」「さらに」「とりわけ」が 1 文章に 3 回以上 → 半分削除。

### A11. 否定並列構文

「〜ではない。〜だ。」が 2 回以上 → 片方を別構文に。

**NG:** 単なるツールではない。パラダイムシフトだ。

**就活向け補足:** 「個人の生産性ツールではなくチームの開発基盤として」のような就活頻出形を含む。

### A12. -ing 的付け足し

データの後に AI が勝手に意義を付け足す。

**NG:** 売上は前年比 120% でした。これは同社の戦略が功を奏していることを示しており、今後の成長が期待されます。
**OK:** 売上は前年比 120% でした。

**就活向け補足:** 「〇〇を改善しました。これは〜を示しています」「〇〇を達成し、〜に貢献しました」が頻出。事実だけで止める。

### A13. 主語の過剰明示

**NG:** このツールは高速です。このツールはセキュリティも強固です。
**OK:** 高速で、セキュリティも強固。

### A14. 「〜することができます」

「設定を変更することができます」→「設定を変更できます」→「設定は変えられます」

### A15. 体温のない結論

「以上のことから〜と考えられます」→ 自分の体温を入れる。
**就活向け補足:** ただし履歴書では「思ってます」「感じてます」は NG。「考えています」「目指しています」程度の温度感に留める（媒体別注意 → SKILL.md）。

## カテゴリ B: 不採用 / 調整

humanizer-ja の以下のパターンは履歴書文脈では採用しない:

- **追従的トーン**（「素晴らしいご質問」等） — 履歴書では発生しない
- **敬語の均一化** — 履歴書はむしろ均一な方が読みやすい
- **「正直〜思ってます」系の voice** — 履歴書媒体に不適合（カバーレターでは可）

## カテゴリ C: 就活ドメイン特化（追加）

### C1. 空疎動詞の連発

以下が **同段落で 2 回以上** 出現したら警戒。「組み込み」は無意識に再出現しやすいので **警戒度を最高ランク** に固定:

「担当」「実現」「貢献」「主導」「推進」「最大化」「横断的に」「一貫して」「**組み込み**」「取り組み」「整備」

**置き換え:** 具体動作に書き換える。**口語動詞（「握る」「渡す」「回す」など）は使わず漢語動詞へ**（C7 参照）。

| NG | OK（漢語動詞中心） |
|---|---|
| 〇〇を担当 | 〇〇を設計・実装・QA まで担当（具体動作を並べる） |
| 〇〇を実現 | 〇〇を達成（数字を添える） |
| 〇〇に貢献 | 〇〇を XX% 改善 |
| 〇〇を主導 | 〇〇の方向性を決定し、チームへ展開 |
| 横断的に推進 | 設計・実装・QA・リリース判断まで責任範囲とした |
| AI を組み込み | Claude Code を設計・実装・レビュー・QA の各フェーズで運用 |
| 〇〇を組み込み | 〇〇を統合 / 連携 / 整備 / 導入 |

### C2. 盛り構文（抽象名詞でドラマ化）

**NG パターン:**
- 「〜という二重の技術的不確実性」
- 「〜という前例のない」
- 「複雑なステークホルダー構造」
- 「〇〇という困難」「〇〇という制約」

**置き換え:** 具体的状況描写に分解する。

**例:**
- ❌ 二重の技術的不確実性
- ✅ コードベースが新規・OEM 連携も初

### C3. チーム規模の盛り

「3 名のチームを率いて」を毎セクションで繰り返さない。**1 つの職歴内で 1 回までが目安**。

### C4. AI 活用節の決まり文句

特に警戒すべき頻出句:

- 「AI を組み込み」
- 「アウトプット最大化」
- 「個人の生産性ツールではなくチームの開発基盤として」
- 「設計・実装・レビュー・QA まで開発プロセス全体に」
- 「少人数チームでのアウトプット量を最大化することにも取り組んできました」

**置き換え:** 具体的に何をしたか（何のフローで・何を回したか・どれくらいの効果が出たか）。

### C5. 数字の単独提示原則

「〇〇を改善」より「〇〇を 70% 短縮」を優先。

- ❌ レスポンスタイムを向上
- ✅ レスポンスタイムを 1 秒以内に短縮（Datadog 計測）

数字の根拠（計測ツール）が出せるとさらに信頼度が上がる。

**数字なしの効果動詞は削除（強化ルール）:**

「向上」「改善」「引き上げ」「最大化」など効果動詞は **数字とセット必須**。数字が出せないなら **効果動詞ごと削除し、打ち手の解像度（誰と・何を・どの頻度で）で完結させる**。

| NG（数字なし効果動詞） | OK（打ち手の解像度で完結） |
|---|---|
| チームの開発生産性を引き上げました | 3 名チームでの開発フローを構築しました |
| パフォーマンスを向上させました | （数字が出せないなら）N+1 クエリを 12 箇所修正し、index を 3 本追加しました |
| 業務効率を改善しました | （数字が出せないなら）週次定例の議題を 5 つに絞り、振り返りを月次に移行しました |

### C6. 文末「ました」連発と体言止めの挿入

職務経歴書では「〜しました」「〜ています」が並びやすい。**3 文連続したら 1 つは体言止め** に変える。これによりモジュール感（voice.md A1）が出る。

**NG（文末連発）:**
- 設計を担当しました。
- 実装を担当しました。
- QA を実施しました。

**OK（体言止め 1 回挿入）:**
- 設計・実装・QA を担当。
- リリース判断まで責任範囲としました。

**履歴書向け体言止めの形:**
- 「〜を統合し、…」→「〜を統合。…」
- 「〜を構築しました」（最後の文に置く）

体言止めは **段落の中盤** で使うと効果的。冒頭や末尾は完全な文の方が読みやすい。

### C7. 動詞のレジスター（口語動詞 NG / 漢語動詞 OK）

履歴書日本語のレジスターは「丁寧 + 名詞中心 + 受身控えめ + 動作主の明示は最小限」。**動詞中心になりすぎると幼稚に見える** ため、書類用の漢語動詞に置換する。

| 口語動詞（NG、書類では使わない） | 漢語動詞（OK、書類向け） |
|---|---|
| 回す | 運用する |
| 握る | 合意形成を担う / 責任を持つ |
| 渡す | 展開する / 引き継ぐ |
| 手を入れる | 担当する / 改修する |
| やってきた | 担ってきました / 従事してきました |
| 棚卸し | 整理 / 洗い出し |
| 整える | 定着させる / 構築する |
| 上げる（数字なし） | 削除 or 別角度の具体性で埋める（C5 参照） |
| 引き上げる（数字なし） | 削除 or 別角度の具体性で埋める（C5 参照） |
| 移転する（社内用語） | 共有する |
| 並べる（議題等） | 整理 / 列挙 |
| 絞る | 選定 / 絞り込み |
| 持っていく | 展開 / 提示 |
| 引き受ける | 担う / 就任する / 引き継ぐ |
| 取り組む（数字なし） | 削除 or 具体動作で代替（C5 参照） |
| 経験してきました | 担当してきました / 従事してきました |

**判断基準:** 「この動詞、社内 Slack なら自然か?」と自問。Slack では自然だが、企業の公式文書（人事評価・社内 wiki）では浮く動詞は履歴書でも浮く。

### C8. 接続詞による文連結より文分割

「並行して」「〜のため」「さらに」のような接続詞でつなぐ前に、**文を分けてモジュール化を試す**（voice.md A1 と整合）。

**NG（接続詞で長文化）:**
> 上長と週次でリスクを並べ、並行してペアプロでドメイン知識を共有し、リファインメントの形を整えました

**OK（文分割でモジュール化）:**
> 上長と週次でリスクを整理し、優先度上位 3 件に絞って対応をチームへ展開しました。ペアプロでドメイン知識を共有し、リファインメントの運用を定着させました

### C9. 同義表現の統一性チェック（v3.1 追加）

**同じ文書内で 1 箇所を直したら、類似表現も同じ扱いで修正する**。例えば「一貫して経験してきました」を直したら、別段落の「横断して経験してきました」も同じパターンとして警戒する。AI 臭の本質は **同じ意味を別表現で繰り返す癖** であり、片方だけ残すと整合性を欠く。

**実行手順:**

1. 1 ヒットを修正したら、その類似フレーズを **必ず grep で全文検索** する
2. 「経験してきました」を直したら → `grep '経験してきました'`
3. 「リーダーシップを引き受け」を直したら → `grep '引き受け|引き受ける'`
4. 「取り組み」を直したら → `grep '取り組み'`

**同義表現グループ（v3.1 時点）:**

| グループ | パターン | 同義表現の例 |
|---|---|---|
| 経験動詞 | 〇〇を / して経験してきました | 一貫して / 横断して / 通して / トータルで / 様々に |
| 引き受け動詞 | リーダーシップを引き受け | リードを引き受け / 旗を振り / 主導権を取り / 旗振り役を担い |
| 取り組み動詞 | 〇〇に取り組みました | 〇〇に注力しました / 〇〇に挑戦しました / 〇〇に着手しました |
| 改善動詞（数字なし） | 改善 | 向上 / 引き上げ / 強化 / 最適化 / 充実化 |
| 主導動詞 | 主導 | リード / 牽引 / 旗振り / 推進 |

**警戒度:** post-check モード（`/humanize <file>`）では、これらのグループを **追加 grep キーワード** として走らせ、見落としを防ぐ。

### C10. JP-EN 全角半角境界スペース（v3.2 追加）

LLM は markdown 整形時に「日本語と英数字の間に半角スペース」を入れる癖がある。視覚的にバランスが取れて見えるが、Yagish のような form 入力では密度が落ちて読みにくく、**人間が書いた日本語履歴書では稀** な典型的 AI 生成シグナル。

**NG 例:**

- ❌ 約 5 年 / 10% 以上 / 2024 年 10 月
- ❌ BCIT で Computer Science を専攻
- ❌ Claude Code を CI に統合

**OK（境界スペースを除去）:**

- ✅ 約5年 / 10%以上 / 2024年10月
- ✅ BCITでComputer Scienceを専攻
- ✅ Claude CodeをCIに統合

**保持する例外:**

- 多語英語フレーズ内のスペース: `Ruby on Rails`, `Claude Code`, `Design Doc`
- 製品名内のスペース: `freee Eラーニング`, `freee 共通決済基盤` は文脈による（ブランド表記に従う）

**検知（hook 自動）:**

`scripts/humanize-check.sh` が `aspects/job/` 配下の編集後に以下を grep:

```
(\p{sc=Han}|\p{sc=Hiragana}|\p{sc=Katakana}) [A-Za-z0-9%]
[A-Za-z0-9%] (\p{sc=Han}|\p{sc=Hiragana}|\p{sc=Katakana})
```

`\p{sc=Han}` は Script（Script_Extensions ではない）指定のため、`〜` (U+301C) の誤マッチを回避できる。

**修正（perl one-liner）:**

```bash
F=path/to/file.md
perl -i -CSD -pe 's/(\p{Han}|\p{Hiragana}|\p{Katakana}|[々ー、。「」『』（）]) ([A-Za-z0-9%])/$1$2/g; s/([A-Za-z0-9%]) (\p{Han}|\p{Hiragana}|\p{Katakana}|[々ー、。「」『』（）])/$1$2/g' "$F"
sed -i 's/ 〜/〜/g; s/〜 /〜/g' "$F"   # 〜 周辺は別途整形
```

**過去の漏れ:** PR #629（2026-04-29）で resume-yagish.md を 8 ペルソナでレビューしたが、文意・構成・重複は見たが境界スペースという「形」のチェックを失念。ユーザー指摘で修正。
