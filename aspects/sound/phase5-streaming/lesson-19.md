## 1. LUFS --- 配信時代のラウドネス基準（15分）

### dB と LUFS の違い

Phase 1-4 では dB（デシベル）で音量を管理してきましたね。配信の世界では、もう1つの基準「LUFS」が重要になります。

```
dB (デシベル):
  → 瞬間的な信号レベルを示す
  → ミキサーのメーターで見る値
  → ピーク（瞬間最大値）を管理するのに適している

LUFS (Loudness Units Full Scale):
  → 人間の聴感に基づいた「体感音量」を示す
  → 一定時間の平均的な音量
  → 配信プラットフォームの基準値として使われる
```

### なぜ LUFS が必要か

```
配信プラットフォームの自動ラウドネス調整:

  あなたの配信 ──→ YouTube ──→ 視聴者
                     │
                 ラウドネス
                 ノーマライゼーション
                     │
                 基準より大きい → 自動で下げる
                 基準より小さい → そのまま（上げない）

  ※ YouTube は基準より大きい音は下げるが、
    小さい音は上げてくれない！
  → 小さすぎる配信は小さいまま視聴者に届く
```

### 各プラットフォームの推奨 LUFS

```
┌────────────────────┬──────────────┬──────────────────┐
│ プラットフォーム    │ 推奨 LUFS    │ 備考              │
├────────────────────┼──────────────┼──────────────────┤
│ YouTube            │ -14 LUFS     │ 超えると自動で     │
│                    │              │ 下げられる         │
├────────────────────┼──────────────┼──────────────────┤
│ Facebook Live      │ -16 LUFS     │ YouTube より       │
│                    │              │ やや控えめ         │
├────────────────────┼──────────────┼──────────────────┤
│ Spotify (参考)     │ -14 LUFS     │ 音楽配信の基準     │
├────────────────────┼──────────────┼──────────────────┤
│ 教会配信の推奨     │ -14〜-16     │ この範囲に収める   │
│                    │   LUFS       │                    │
└────────────────────┴──────────────┴──────────────────┘
```

> **田中メモ:** 「-14 から -16 LUFS」を覚えてください。この範囲なら YouTube でも Facebook でも適切な音量で配信されます。-14 より大きいと YouTube に自動で下げられ、-16 より小さいと視聴者に「音が小さい」と思われます。

### ラウドネスメーターの使い方

```
ラウドネスメーター画面の見方:

  ┌─────────────────────────────────┐
  │  Integrated LUFS:  -15.2       │ ← 配信全体の平均。最重要
  │  Short-term LUFS:  -13.8       │ ← 直近3秒の平均
  │  Momentary LUFS:   -11.4       │ ← 瞬間的な値
  │  True Peak:        -1.2 dBTP   │ ← ピーク値
  │                                 │
  │  ■■■■■■■■■■■■□□□□ -15.2 LUFS  │ ← バーメーター
  │              ↑                   │
  │          ここを -14〜-16 に     │
  └─────────────────────────────────┘

  無料のラウドネスメーター:
  - Youlean Loudness Meter（プラグイン、無料版あり）
  - dpMeter（スタンドアロン、無料）
  - OBS のラウドネスメータープラグイン
```

### リミッターの設定（安全ネット）

```
配信バスの出力にリミッターを挿す:

  配信 Aux Bus ──→ リミッター ──→ 配信 PC

  リミッターの設定:
  ┌─────────────────────────────┐
  │ Threshold:  -1.0 dBFS      │ ← これ以上の信号を通さない
  │ Release:    100-200ms       │ ← 自然に戻す速度
  │ Ceiling:    -1.0 dBFS      │ ← 出力の絶対上限
  └─────────────────────────────┘

  ※ -1.0 dBFS にする理由:
    0 dBFS ぴったりだとエンコード時にクリップする可能性
    -1.0 dBFS なら安全マージンが確保できる
```

> **田中メモ:** リミッターは「保険」です。通常の運用では作動しないレベルに設定しておき、突発的な大音量（マイクを落とした、ハウリングが起きた等）から配信リスナーの耳を守ります。

## 2. モニタリング体制の構築（10分）

### 理想的なモニタリング体制

```
【3段階モニタリング体制】

  第1段階: ミキサーのメーター（常時）
  ┌──────────────────┐
  │ 配信バスのレベル   │ → ピーク管理
  │ メーターを常時監視 │ → 赤に振れたら即対応
  └──────────────────┘

  第2段階: ヘッドホンモニター（常時）
  ┌──────────────────┐
  │ 配信バスを         │ → 音質・バランス確認
  │ ヘッドホンで聴く   │ → ノイズ・歪みの即時発見
  └──────────────────┘

  第3段階: 実配信確認（定期的）
  ┌──────────────────┐
  │ スマホ/タブレットで │ → 視聴者と同じ体験を確認
  │ 実際の配信を確認   │ → 遅延あり（5-30秒）
  └──────────────────┘
```

### 配信と会場の遅延問題

```
配信の遅延（レイテンシー）:

  会場(リアルタイム) ──→ エンコード ──→ アップロード ──→ サーバー ──→ 視聴者
                          +1-2秒        +2-5秒          +1-3秒       +1-5秒

  合計遅延: 約 5〜30秒

  これが問題になる場面:
  - 会場でスマホ確認 → スマホから音が漏れて会場に遅れた音が混ざる
  - 対策: 確認用デバイスは必ずイヤホン使用 or 音量ゼロで映像のみ確認
```

### 2人体制の推奨

```
役割分担:

  FOH エンジニア          配信モニター担当
  ┌──────────────┐       ┌──────────────┐
  │ 会場の音を管理 │       │ 配信の音を監視 │
  │ Main L/R      │       │ 配信 Aux Bus  │
  │ フロアモニター │       │ ヘッドホン    │
  │               │       │ + スマホ確認  │
  └──────────────┘       └──────────────┘

  1人で兼任する場合:
  → ヘッドホンの片耳を外して会場音も聴く
  → スマホ確認は賛美と説教の切り替え時に行う
```

## 3. 配信でよくある音声トラブル（15分）

### トラブル 1: 配信に音声が出ない

```
【トラブルシューティング: 音が出ない】

  配信に音声がない！
      │
      ▼
  ミキサーの配信バスメーターは振れている？
      │
      ├── NO → ミキサー側の問題
      │         │
      │         ├── 配信バスのマスターがミュート？ → 解除
      │         ├── 各chの配信Aux送りが0？ → 上げる
      │         └── バスのアサインが間違い？ → 確認
      │
      └── YES → ミキサーからPCへの経路の問題
                │
                ▼
          PC の入力メーターは振れている？
                │
                ├── NO → 接続の問題
                │         │
                │         ├── ケーブルが抜けている？ → 接続確認
                │         ├── PC のオーディオ入力設定？ → 確認
                │         └── ドライバーの問題？ → 再起動
                │
                └── YES → 配信ソフトの問題
                          │
                          ├── OBS の音声ソースがミュート？ → 解除
                          ├── 音声ビットレートが0？ → 設定確認
                          └── 配信先に正しく送信されている？
                              → 配信プラットフォームのダッシュボード確認
```

### トラブル 2: 音声にクリップ・歪みがある

```
【トラブルシューティング: 音が歪む】

  配信の音が歪んでいる！
      │
      ▼
  どの段階で歪んでいる？
      │
      ├── ミキサーのバスメーターが赤 → ミキサー内でクリップ
      │   │
      │   ├── 配信バスのマスターを下げる
      │   ├── 原因チャンネルの Aux 送りを下げる
      │   └── リミッターが未設定なら挿す
      │
      ├── PC の入力で赤 → ミキサー出力が大きすぎる
      │   │
      │   ├── ミキサーの出力レベルを下げる
      │   └── PC の入力ゲインを下げる
      │
      └── 配信ソフトのメーターが赤 → ソフト内でクリップ
          │
          ├── OBS のゲインフィルターを確認
          └── 出力レベルを調整
```

### トラブル 3: 音声と映像がずれている

```
【トラブルシューティング: 音ズレ】

  音と映像がずれている！
      │
      ▼
  音が映像より早い？遅い？
      │
      ├── 音が早い（口の動きより先に声が聞こえる）
      │   │
      │   └── 映像処理に時間がかかっている
      │       → 配信ソフトで音声に遅延（ディレイ）を追加
      │       → OBS: 音声ソース → フィルタ → 同期オフセット
      │
      └── 音が遅い（口が動いてから声が聞こえる）
          │
          └── 音声処理に時間がかかっている
              → 音声の処理チェーンを短くする
              → バッファサイズを小さくする
              → エフェクト処理を減らす

  同期の確認方法:
  1. 手を叩く映像を撮る
  2. 配信で手を叩く音と映像のタイミングを確認
  3. ずれている分だけオフセットを調整
```

### トラブル 4: 配信にエコーが入る

```
【トラブルシューティング: エコー】

  配信にエコーが聞こえる！
      │
      ▼
  エコーの原因を特定:
      │
      ├── 配信 PC のスピーカーが ON
      │   → 配信 PC から出た音を会場マイクが拾っている
      │   → 対策: 配信 PC のスピーカーを必ずミュート
      │
      ├── 確認用スマホ/タブレットの音漏れ
      │   → スマホの音が会場マイクに入っている
      │   → 対策: 確認デバイスはイヤホン必須
      │
      └── 会場内に配信を映すモニターがある
          → モニターのスピーカーから音が出ている
          → 対策: 会場モニターの音声はミュート

  ★ 鉄則: 配信の音を会場のスピーカーから
           絶対に出さない！
```

## 4. 回線品質と音声（5分）

### インターネット帯域と音質の関係

```
配信に必要なアップロード帯域:

  映像（1080p）:  4-6 Mbps
  音声（256kbps）: 0.256 Mbps
  合計:           約 5-7 Mbps

  推奨アップロード速度:
    最低: 10 Mbps（配信速度の1.5〜2倍）
    推奨: 20 Mbps 以上

  確認方法:
    fast.com や speedtest.net でアップロード速度を計測
    → 礼拝前に毎回確認する習慣をつける
```

### 回線が不安定な場合の対策

```
帯域が足りない場合の優先順位:

  1. 映像品質を下げる（1080p → 720p）
     → 音質はそのまま維持。音が最優先！
  2. フレームレートを下げる（30fps → 24fps）
  3. 音声ビットレートは最後まで下げない
     → 128kbps 以下は音質劣化が目立つ

  音声ビットレートの目安:
    256 kbps ステレオ  最高品質
    192 kbps ステレオ  高品質（推奨）
    128 kbps ステレオ  最低限
     64 kbps モノラル  非推奨（音楽がつらい）
```

> **田中メモ:** 「映像は少し粗くても礼拝は成立しますが、音声が途切れたり歪んだりすると礼拝への集中が一気に崩れます。帯域が厳しいときは、映像を犠牲にしてでも音声品質を守ってください。」

## 5. バックアッププラン（5分）

### 配信が落ちた場合の対応

```
配信障害対応フロー:

  配信が止まった！
      │
      ▼
  ┌──────────────────────────────────────┐
  │ 1. まず落ち着く（会場の礼拝は続行）  │
  │    → 配信トラブルで礼拝を止めない    │
  └──────────────────┬───────────────────┘
                     │
  ┌──────────────────▼───────────────────┐
  │ 2. 原因を切り分ける（30秒以内）       │
  │    → インターネット接続を確認          │
  │    → 配信ソフトの状態を確認            │
  │    → プラットフォームの障害情報を確認  │
  └──────────────────┬───────────────────┘
                     │
  ┌──────────────────▼───────────────────┐
  │ 3. 復旧を試みる                       │
  │    → 配信ソフトで「配信開始」を再押下  │
  │    → ダメなら PC を再起動              │
  │    → それでもダメならバックアップ手段   │
  └──────────────────┬───────────────────┘
                     │
  ┌──────────────────▼───────────────────┐
  │ 4. バックアップ配信手段                │
  │    → スマホで直接配信（Instagram等）   │
  │    → 録音は継続（後からアップ可能）    │
  └──────────────────────────────────────┘
```

### 事前に準備しておくこと

```
バックアップ準備チェックリスト:

  □ 録音を配信とは別系統で行う
    → 配信が落ちても録音は残る
    → 後からアーカイブとしてアップロードできる

  □ スマホ配信の準備
    → YouTube アプリで配信できるようアカウント設定済み
    → 三脚とスマホホルダーを用意

  □ 配信プラットフォームの管理者アカウント
    → 配信担当者がログインできる状態にしておく

  □ トラブル時の連絡手段
    → 配信チームの LINE グループ等
```

## 教会での応用

- **配信品質チェック日を設ける:** 月に1回、礼拝の配信アーカイブをラウドネスメーターで計測し、-14〜-16 LUFS に収まっているか確認する
- **トラブル対応マニュアルを作る:** 今日学んだフローチャートをA4用紙1枚にまとめて、配信卓の横に貼っておく。本番中に慌てて思い出す必要がなくなります
- **配信テスト配信:** 月に1回、礼拝のない日にテスト配信を行い、全経路が正常に動作するか確認する。限定公開にすれば外部には見えません
- **視聴者フィードバック:** 配信で礼拝に参加している方に定期的に音質の感想を聞く。「聞きやすかった？」「音量はちょうどよかった？」
