# 勉強 - 2026-03-17

## 🎯 今日の目標・疑問

設計の規模感をどう素早く見積もるか。面接でBack-of-Envelopeを使いこなすための計算パターンを理解する。

## 📝 ノート

### Back-of-Envelope 見積もり計算

**一言で言うと（ファインマン式）**
設計の規模感（QPS・ストレージ・キャッシュ）を大まかに把握するための概算計算。正確さより「オーダー感を掴む」ことが目的。

**計算式まとめ**

QPS（1秒あたりのリクエスト数）:
```
平均QPS  = DAU × 1人あたりのリクエスト数/日 ÷ (24 × 3600)
ピークQPS = 平均QPS × 2（安全係数。トラフィック特性による）
```

ストレージ:
```
ストレージ = DAU × 1操作あたりのデータ容量 × 操作回数/日 × 365 × 保存期間（年）
```

キャッシュ容量:
```
キャッシュ容量 = DAU × 1人あたりの読み取りデータ量 × 20%
```
パレートの法則: アクセスの80%は上位20%のデータに集中 → ホットデータ20%だけキャッシュすれば十分

**よく使う単位換算**

| 単位 | バイト数 |
|------|---------|
| KB | 10³ |
| MB | 10⁶ |
| GB | 10⁹ |
| TB | 10¹² |
| PB | 10¹⁵ |

**面接での使い方**
- 数字の正確さより「計算プロセスを見せること」が重要
- 仮定は必ず口に出して宣言する（「〜と仮定します」）
- ピークQPSの2倍は安全係数。SNSや季節性があるサービスは3〜5倍も考慮

## 🔑 キーワード

- **DAU（Daily Active Users）** — 1日あたりのアクティブユーザー数。QPS計算の起点
- **QPS（Queries Per Second）** — 1秒あたりのリクエスト数
- **ピークQPS** — 平均QPSの2倍が基本
- **パレートの法則（80/20）** — アクセスの80%は上位20%のデータに集中
- **Back-of-Envelope** — ざっくり見積もり。正確さより桁感を掴むのが目的

## 💡 まとめ

1. Back-of-Envelopeは「正確な数字を出す」のではなく「桁感を掴む」ための計算
2. QPS = DAU × リクエスト数/日 ÷ 86400、ピークは×2
3. キャッシュはホットデータ20%だけで全アクセスの80%をカバーできる

## ❓ 残った疑問・次回へ

- 面接でこの計算をスムーズに口頭でできるようになりたい（練習が必要）
- ピークQPSを何倍にするかの判断基準（サービスの特性による）
