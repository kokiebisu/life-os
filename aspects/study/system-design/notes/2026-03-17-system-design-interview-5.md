# 勉強 - 2026-03-17

## 🎯 今日の目標・疑問

シャーディングのジョイン問題とその対処法を理解する。分散DBで避けられないトレードオフを把握する。

## 📝 ノート

### DBスケーリング（シャーディング）

**シャードとは**
テーブルの**行のグループ**。1つの行は必ず1つのシャードに属する。

```
usersテーブル全体:
  シャードA（サーバA）→ user_id 1〜1000万 の行
  シャードB（サーバB）→ user_id 1000万〜2000万 の行
```

- 1つのシャード（サーバ）に**複数のテーブル**が入ることもある
- 1つのテーブルが**複数のシャードにまたがる**こともある（巨大テーブルの分割）

**ジョイン問題**
通常のJOINはデータが同じサーバにある前提。シャーディングするとデータが別サーバに分散するため、**クロスシャードJOINが難しくなる**。

**対処法まとめ**

| 状況 | 対処法 |
|------|--------|
| 関連データの更新が少ない | 非正規化（user_nameをordersに持たせるなど） |
| 関連テーブルを同じキーで分割できる | シャードキー設計（user_idで揃える） |
| 大規模システムでアプリ側を単純にしたい | ミドルウェア（Vitess/Citus） |
| 複雑な分析クエリが必要 | OLAP / データウェアハウス |

**非正規化のトレードオフ**
- メリット: JOINが不要になる
- デメリット: 元データが変わった時に全行を更新しないといけない
- → **更新が少ないデータ**に向いている

**シャードキー設計で同じシャードに寄せる**
user_idをシャードキーにする → users も orders も同じ user_id のデータは同じシャードに入る → JOINがシャード内で完結する。

**ミドルウェア（シャードルーター）**
アプリからは1回のクエリに見えるが、内部で複数シャードに並列クエリを投げてマージする。代表例: **Vitess**（MySQL）、**Citus**（PostgreSQL）

**分析用DBを別途持つ（OLAP / データウェアハウス）**
更新系（OLTP）とは別に、全データを集めた読み取り専用DBを用意する。複雑なJOINが必要な分析クエリはこちらで実行する。

## 🔑 キーワード

- **シャード** — テーブルの行のグループ。複数サーバーに分散
- **シャードキー** — どのシャードに入るか決めるキー（例: user_id）
- **クロスシャードJOIN** — 別サーバーにまたがるJOIN。パフォーマンス問題が発生
- **非正規化** — JOINをなくすためにデータを冗長に持つ。更新頻度が低いデータ向け
- **Vitess / Citus** — シャードルーターの代表例（MySQL / PostgreSQL向け）
- **OLTP vs OLAP** — OLTP: 更新系 / OLAP: 分析系。複雑なJOINはOLAP側で
- **データウェアハウス** — 全データを集めた読み取り専用DB。複雑な分析に対応

## 💡 まとめ

1. シャーディングするとクロスシャードJOINが難しくなる。設計段階でキーを揃えて回避するのが基本
2. 非正規化は更新が少ないデータに有効。更新頻度が高いと更新コストが爆発する
3. 分析クエリはOLTP（シャーディングDB）では無理。データウェアハウスを別に持つ
