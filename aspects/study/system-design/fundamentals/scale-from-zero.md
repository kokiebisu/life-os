# ゼロからスケールへ

## 🧒 一言で言うと（ファインマン式）
ユーザーが増えるにつれて、一台のサーバーから始まり、ロードバランサー・DB レプリケーション・キャッシュ・CDN・シャーディングと段階的に設計を進化させていく話。

## スケールの進化フロー

### Phase 1: 単一サーバー
- アプリサーバーと DB が同一マシンに共存
- クライアント → DNS → Public IP → Web サーバー → HTML を返す

### Phase 2: スケールアップ vs スケールアウト
- スケールアップ（垂直）: 高コスト・上限あり・冗長性なし → 採用しない
- スケールアウト（水平）: ロードバランサーを導入してWebサーバーを複数台に
- DNS にはロードバランサーの Public IP を登録
- Web サーバーは Private IP に切り替え、セキュリティを確保

### Phase 3: DB レプリケーション（マスター/スレーブ）
- マスター: 書き込み専用
- スレーブ: 読み取り専用（マスターから複製）
- **メリット:** フェイルセーフ・読み書きを独立してスケールアウト可能

**フェイルオーバーの挙動:**
| 障害箇所 | 動作 |
|----------|------|
| スレーブ障害 | 他スレーブにリダイレクト → 新スレーブ立ち上げ後に復帰 |
| マスター障害 | スレーブ1台がマスターに昇格 → スクリプトで未反映データを補完 → 新スレーブ追加 |

### Phase 4: キャッシュ
- 頻繁な読み取りの結果をメモリに保存 → DB 負荷↓・レイテンシ↓
- **注意:** 読み取りが少ない箇所には費用対効果が薄い

### Phase 5: CDN
- 静的コンテンツを地理的に近いサーバーから配信
- Web サーバーへのアクセスを削減

### Phase 6: ステートレス化
- セッションデータを Web 層ではなく専用データストアで管理
- スティッキーセッション不要 → ロードバランサーが自由にルーティング可能

### Phase 7: データセンター（マルチリージョン）
- 複数地域にサーバー群を配置 → 地理的冗長性・レイテンシ改善

### Phase 8: メッセージキュー
- コンポーネント間を疎結合にする
- 重い処理のボトルネックを非同期化

### Phase 9: DB シャーディング
- 大きなテーブルの行を複数のシャードに分散 → 読み書きが該当シャードのみに限定
- **課題:**
  - シャーディングキーの選び方が難しい（不均等分散のリスク）
  - シャード容量超過時の再パーティション作業
  - クロスシャード JOIN の困難さ（ACID 保証が難しい）

## SQL vs NoSQL の選択基準

**NoSQL を選ぶとき:**
1. **Eventual Consistency で許容できる**（ACID不要）— SNSのいいね数、閲覧数など一瞬の不整合が問題ない場合
2. 超低レイテンシが必要
3. データが非構造化・スキーマが流動的
4. **1台のマシンに収まらないデータ量** — スレーブも同じデータを全量持つため、SQLレプリケーションでは根本解決にならない
5. **SQLレプリケーションの限界を超えるQPS** — データ量が巨大だとマスター→スレーブの同期ラグが大きくなる
6. シリアライズ済みデータ（JSON等）をそのまま格納したい

**SQL を選ぶとき:**
1. ACID保証が必要（金融・決済等）
2. データ間の関係が複雑（JOINが必要）
3. スキーマが安定している

> 「読み取りが多い = NoSQL」は誤り。**キャッシュ + レプリカで捌ける規模ならSQLで十分。**
> 判断の核心は「データ量がマシン1台を超えるか」「ACID（整合性）が必要か」。

## トレードオフ
| 手法 | メリット | デメリット |
|------|----------|------------|
| スケールアウト | 冗長性・上限なし | 複雑さ増加 |
| DB レプリケーション | フェイルセーフ・読み書き分離 | フェイルオーバーの複雑さ |
| キャッシュ | 高速・DB 負荷軽減 | 整合性管理が必要 |
| シャーディング | 水平スケール可能 | キー設計・JOIN・ACID が難しい |
