# DBスケーリング

## シャーディング（水平分割）

### シャードとは

テーブルの**行のグループ**。1つの行は必ず1つのシャードに属する。

```
usersテーブル全体:
  シャードA（サーバA）→ user_id 1〜1000万 の行
  シャードB（サーバB）→ user_id 1000万〜2000万 の行
```

- 1つのシャード（サーバ）に**複数のテーブル**が入ることもある
- 1つのテーブルが**複数のシャードにまたがる**こともある（巨大テーブルの分割）

---

### ジョイン問題

通常のJOINはデータが同じサーバにある前提で設計されている。
シャーディングするとデータが別サーバに分散するため、**クロスシャードJOINが難しくなる**。

```
シャードA: orders テーブル
シャードB: users テーブル

→ orders JOIN users はサーバをまたぐ → ネットワーク通信が発生
```

---

### 対処法

#### 1. 非正規化（更新が少ないデータ向け）

JOINが不要になるよう、関連データを同じテーブルに冗長に持つ。

```
【正規化】
orders(order_id, user_id)  +  users(user_id, name)
→ name を取得するにはJOINが必要

【非正規化】
orders(order_id, user_id, user_name)
→ user_name を orders に持たせてJOIN不要に
```

**トレードオフ:** user_name が変わったとき orders の全行を更新しないといけない。
→ **更新が少ないデータ**に向いている（更新頻度が高いと更新コストが爆発する）

#### 2. シャードキー設計で同じシャードに寄せる

関連するテーブルを同じシャードキーで分割することで、JOINを同一シャード内に収める。

```
user_id をシャードキーにする
→ users も orders も同じ user_id のデータは同じシャードに入る
→ JOINがシャード内で完結する

シャードA: user_id 1〜1000万 の users + orders + reviews
シャードB: user_id 1000万〜  の users + orders + reviews
```

**限界:** 異なるシャードキーでJOINしたいときは結局難しい。

#### 3. アプリ側でマージ

各シャードに個別クエリを投げ、アプリ側で結果を結合する。

```
アプリ → シャードA に クエリ
アプリ → シャードB に クエリ
アプリ側で結果をマージ
```

#### 4. ミドルウェア（シャードルーター）

アプリからは1回のクエリに見えるが、内部で複数シャードに並列クエリを投げてマージする。

```
アプリ → ルーター → シャードA, シャードB（並列）→ 結果マージ → アプリ
```

代表例: **Vitess**（MySQL）、**Citus**（PostgreSQL）

#### 5. 分析用DBを別途持つ（OLAP / データウェアハウス）

更新系（OLTP）とは別に、全データを集めた読み取り専用DBを用意する。
複雑なJOINが必要な分析クエリはこちらで実行する。

---

### まとめ：どの対処法を使うか

| 状況 | 対処法 |
|------|--------|
| 関連データの更新が少ない | 非正規化 |
| 関連テーブルを同じキーで分割できる | シャードキー設計 |
| 大規模システムでアプリ側を単純にしたい | ミドルウェア（Vitess等） |
| 複雑な分析クエリが必要 | OLAP / データウェアハウス |
