# /kondate コマンド設計書

**Date:** 2026-03-17
**Status:** Approved

---

## 概要

`/kondate` は、冷蔵庫の在庫ベースで作り置きを考慮した複数食分の献立を AI が自動提案し、ユーザー承認後に daily ファイルと Notion meals DB を一括登録するコマンド。

### `/meal` との棲み分け

| コマンド | 用途 |
|---------|------|
| `/kondate` | **計画**: これから食べる献立を提案・登録 |
| `/meal` | **記録**: 食べた食事を記録（未登録 or 差し替え） |

---

## フロー

### Step 1: 状況把握

```bash
TZ=Asia/Tokyo date "+%Y-%m-%d %H:%M %a"
```

- `aspects/diet/fridge.md` を読んで生鮮食品の在庫を把握
- `aspects/diet/pantry.md` を読む（常備調味料は在庫チェック不要 — 常にあるものとして扱う）
- 直近の `aspects/diet/daily/YYYY-MM-DD.md` を確認（登録済み食事の有無）
- Notion events DB で今後のイベント（外食・飲み会）を確認

状況把握が終わったら、**除外される食事枠をユーザーに提示してから食数を聞く:**

```
〇日分で計画します。
・火曜夜: events DB に「飲み会」があるため除外

何食分計画しますか？（例: 朝3・昼0・夕2 = 5食）
```

### Step 2: 献立を自動提案

食数の回答を受けたら、以下の優先順位で献立を構成する:

1. **作り置き向き食材を特定** — fridge.md の中で量が多いもの（肉・キャベツ等）を複数食にまたがって使い切る計画を立てる
2. **実在レシピをプラットフォームから検索** — 以下の優先順で使用するプラットフォーム:
   - [クラシル](https://www.kurashiru.com/) — メイン（普段の献立）
   - [白ごはん.com](https://www.sirogohan.com/) — 和食基本
   - [Nadia](https://oceans-nadia.com/) — アレンジ
   - [DELISH KITCHEN](https://delishkitchen.tv/) — 簡単メニュー
3. **NG食材を除外** — トマト、マヨネーズ、ケチャップ、マスタード（`profile/health.md` 参照）
4. **材料の完全リストを提示** — 調味料を含む全材料を1人前換算で表示

**在庫にない食材の扱い（厳守）:**

原則として `fridge.md` にある食材のみで献立を組む（diet CLAUDE.md「献立は在庫ベース」ルール準拠）。ただし、不足食材が少量の調達で済む場合は「⚠️ 不足」として提案に含め、承認時に自動で買い出し登録する（後述 Step 4d）。

**`pantry.md` の常備調味料は在庫チェック不要** — 醤油・ごま油・みりん等は常備品として存在する前提で献立に組む。

### Step 2 の提案フォーマット

```
【月 朝食】オートミール + 蜂蜜
  材料: オートミール 40g、蜂蜜 大さじ1
  推定: ~300kcal
  在庫: ✅ 全材料あり

【月 夕食】豚キャベツ蒸し（クラシル）
  材料: 豚こま 200g、キャベツ 1/4個、生姜 1片、
        酒 大さじ1、醤油 大さじ1、ごま油 小さじ1
  推定: ~480kcal
  在庫: ✅ 全材料あり

【火 夕食】豚の生姜焼き + 玄米（白ごはん.com）※作り置き豚こま使用
  材料: 豚こま 150g（作り置き）、玄米パック 1食、
        醤油 大さじ1.5、みりん 大さじ1、酒 大さじ1、砂糖 小さじ1
  推定: ~550kcal
  在庫: ⚠️ 不足: 長ねぎ（承認時に買い出し登録します）
```

レシピが見つからない場合は別プラットフォームで再検索し、それでも見つからない場合のみシンプルな定番メニュー（オートミール等）に差し替える。

### Step 3: ユーザー承認 or 修正

- 「OK」「LGTM」→ Step 4 へ
- 「〇〇を変えて」「火曜の夕食なしにして」→ 該当部分のみ差し替えて再提案
- 修正は何度でも対応する

### Step 4: 一括登録

承認後、全食事を以下の手順で登録する:

#### 4a. Notion meals DB に登録

```bash
# 自炊（レシピ自動生成あり）
bun run scripts/notion-add.ts --db meals --title "メニュー名" --date YYYY-MM-DD --start HH:MM --end HH:MM
```

`notion-add.ts` が内部で `validate-entry.ts`（重複チェック）と `notion-recipe-gen.ts`（レシピ生成）を自動実行するため、別途これらを手動実行しない。

食事時間のデフォルト:
- 朝食: 08:00〜09:00
- 昼食: 12:00〜13:00
- 夕食: 19:00〜20:00

類似エントリが検出された場合（`notion-add.ts` が終了コード 1 を返す）はユーザーに確認してから登録する。

#### 4b. daily ファイル作成・更新

`aspects/diet/daily/YYYY-MM-DD.md` を作成または更新。フォーマットは既存の daily ファイル（例: `aspects/diet/daily/2026-03-06.md`）と同一のセクション見出し + 箇条書き形式:

```markdown
# YYYY-MM-DD（曜日）

## 朝食 08:00-09:00
メニュー名
- 材料1 量
- 材料2 量
- ~XXX kcal

## 昼食 12:00-13:00
メニュー名
- 材料1 量
- ~XXX kcal

## 夕食 19:00-20:00
メニュー名
- 材料1 量
- 材料2 量
- ~XXX kcal

**合計: ~XXXX kcal**
```

注: 食事をスキップした枠は「—」と記載。外食の場合は「外食（詳細）」と記載。

#### 4c. fridge.md は変更しない

消費時にイベント駆動で減算する既存ルール通り。`/kondate` では触らない。

#### 4d. 不足食材がある場合: 買い出しリスト更新

⚠️ 不足食材が1つ以上あった場合、`notion-grocery-gen.ts` で買い出しリストを再生成する:

```bash
bun run scripts/notion-grocery-gen.ts --date YYYY-MM-DD
```

手動で買い出しページを編集しない（フォーマット崩れ防止）。

### Step 5: 報告

```
9食分の献立を登録した。
月: 朝 オートミール(300) / 昼 — / 夕 豚キャベツ蒸し(480)
火: 朝 オートミール(300) / 昼 — / 夕 豚生姜焼き+玄米(550)
水: 朝 オートミール(300) / 昼 — / 夕 豚キャベツ炒め(460)
合計推定: ~2,390kcal/日平均
✅ 買い出しリスト更新済（長ねぎ追加）
```

## 制約・ルール

- **NG食材チェック必須**: `profile/health.md` 参照（トマト・マヨネーズ・ケチャップ・マスタード）
- **在庫ベース優先**: `fridge.md` にある食材から献立を組む。不足食材は「⚠️ 不足」として提示し、承認時に自動で買い出し登録
- **pantry.md の常備調味料は在庫チェック不要**: 常にあるものとして扱う
- **タイムゾーン**: Notion 登録時は必ず `+09:00` を付ける（`notion-add.ts` が処理）
- **レシピは実在プラットフォームから**: 自作レシピ禁止。優先順: クラシル → 白ごはん.com → Nadia → DELISH KITCHEN
- **材料は完全網羅**: 調味料含め全材料を1人前換算で提示
- **イベント連動**: events DB にある食事時間帯は「外食」として献立から除外
- **買い出しリストは手動編集しない**: `notion-grocery-gen.ts` で再生成する

## 既存コマンドとの連携

- **`/meal`**: 登録済みの食事枠に `/kondate` で重複登録しない（`notion-add.ts` 内部チェックで防止）
- **`/cleanup`**: 過去日の未完了エントリは `/cleanup` で処理
- **`/calendar`**: events DB の確認に利用（Step 1）
